# Git/GitHub の運用
このドキュメントでは、TERA プロジェクトにおける Git および GitHub の運用方法について説明します。

## ブランチ戦略
常に存在するブランチは以下の二つです。
- `main`: 本番環境にデプロイされる安定版コードを保持します。**直接コミットしないでください**。
- `develop`: 開発中のコードを統合するブランチです。新しい機能や修正はここにマージされます。

上記二つのブランチに加えて、以下のようなブランチを作成して運用します。
- `<人名>/<内容>`: 作業するためのブランチです。誰が何をするのかが分かる名前にしてください（例: `suzuki3jp/implement-login`）。

## コミットメッセージのルール
コミットメッセージのフォーマットは厳格には決めていません。ただし、以下の点に注意するとより良いです。

- 基本的には日本語で書く
- prefix をつけて目的を明確にする
  - `fix: ` バグ修正
  - `feat: ` 新機能の追加
  - `docs: ` ドキュメントの変更
  - `style: ` フォーマットの変更（コードの動作に影響しないもの）
  - `refactor: ` リファクタリング
  - `chore: ` 細かな変更
- 何を変更したのかを説明するとともに、なぜその変更が必要なのかも簡潔に書く

## CI/CD
GitHub Actions を使用しています。
- PR で常に Lint, Format が実行されます。
- Vercel によってすべてのコミットでプレビューがデプロイされます。
- `main` ブランチにマージされると自動的に本番環境にデプロイされます。
- `develop` ブランチにマージされると自動的に開発環境にデプロイされます。
